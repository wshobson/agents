### ❌ **Factual Error — CONFIRMED**

**Claim (ADR 2, Security section):** "2,632 lines across 79 files ≈ 158 KB uncompressed. The '~2.6MB' figure circulating in discussion is off by roughly 16×."

**Verification:**
```bash
$ wc -l plugins/*/GEMINI.md | tail -1
2389 total  # NOT 2,632
```

**Discrepancy:** Off by 243 lines (9.2% error)

**Root cause:** Unknown. The line count may have been reduced since the ADR was written, or was miscalculated initially.

**Impact:** Minor — the point about file size being negligible is still correct. The actual numbers:
- **2,389 lines** (not 2,632)
- **79 files**
- **Estimated size:** ~142 KB uncompressed (not 158 KB) — still negligible

**Recommendation:** Update ADR 2 to cite the correct line count. The security/maintenance argument remains valid.

### ✅ **Additional Source Claims Verified**

| Claim | Source Cited | Verification | Status |
|-------|--------------|--------------|--------|
| "`@{path}` resolves relative to CWD, not extension dir" | https://geminicli.com/docs/cli/custom-commands/ | **VERIFIED**: "Workspace-aware: The command searches for the path in the current directory and any other workspace directories. Absolute paths are allowed if they are within the workspace." | ✅ Verified |
| "File injection: `@{path/to/file.txt}`" | https://geminicli.com/docs/cli/custom-commands/ | **VERIFIED**: Full documentation of `@{...}` syntax and behavior present | ✅ Verified |

### ✅ **Conclusion on Decision 3 Reasoning**

ADR 3's claim that `@{path}` resolves to workspace directories (not extension root) is **correct and verified**. The decision to use self-contained prompts is sound. Extension-relative inclusion (`@{~/.gemini/extensions/<name>/plugins/...}`) would require:
- Absolute paths to user system directories (poor UX, non-portable)
- Or extension-specific path resolution (not documented, likely not implemented)

The ADR's analysis is correct.

### ❌ **CRITICAL FLAW — Power User Auto-Loading Claim (INVALID)**

**Claim (ADR 2, Option 3, line 62):**
> "Users who clone and `cd plugins/security-scanning/` get the GEMINI.md scoped context automatically via subdirectory scanning"

**Verification:**

Gemini CLI searches for `.gemini/GEMINI.md` files (in a `.gemini/` subdirectory), not arbitrary `GEMINI.md` files at the repository root.

```
Actual structure:          Expected for auto-loading:
plugins/security-scanning/  plugins/security-scanning/
  ├── GEMINI.md            ├── .gemini/
  ├── agents/              │   └── GEMINI.md
  └── ...                  ├── agents/
                           └── ...
```

**Scenario test:**
```bash
$ cd ~/.gemini/extensions/claude-code-workflows/plugins/security-scanning
$ gemini  # Starts Gemini CLI in this directory

Gemini CLI searches for: ./.gemini/GEMINI.md
Actual file location:    ./GEMINI.md

Result: ❌ File NOT loaded
```

**Impact:** The "developer/power user" benefit claimed in ADR 2 is **invalid**. The per-plugin GEMINI.md files provide NO automatic context loading, even in developer workflows. They function only as human-readable documentation in the repository.

**Corrected ADR 2:** Updated to remove false "developer workflow" claim. Files now documented as "human reference only."

**Consequence:** Option 3's justification is weakened. The decision to keep 79 GEMINI.md files is now purely about:
- Human readability (developers can browse plugin docs in the repo)
- Future extensibility (if extension-relative `@{path}` is added)

The "dead weight" concern raised earlier is MORE valid than originally assessed.

**Claim (ADR 1):** "100 Gemini slash commands mirroring Claude Code's full command surface"

**Verification:**
- Claude Code commands: 100 `.md` files
- Gemini TOML commands generated: 115 `.toml` files
- Difference: +15 files

**Cause:** Mix of nested and flat-root commands. Examples:
- Nested: `commands/backend-development/feature-development.toml` → `/backend-development:feature-development`
- Root: `commands/c4-architecture.toml` → `/c4-architecture`
- Root: `commands/backend-development.toml` → `/backend-development` (also has nested commands)

**Status:** ⚠️ Minor discrepancy — the extra 15 files are likely:
1. Plugin-level shortcuts at root (e.g., `/backend-development` + `/backend-development:feature-development`)
2. Multi-command plugin aggregators

This is not a problem but should be documented for clarity.

**Recommendation:** Clarify in ADR 1 whether 100 or 115 commands will be exposed to users.
