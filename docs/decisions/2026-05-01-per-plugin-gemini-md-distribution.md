# ADR: Per-Plugin GEMINI.md — Distribution and Purpose

**Date:** 2026-05-01  
**Status:** Accepted  
**Deciders:** Project maintainers

---

## Context

Phase B created 79 per-plugin `GEMINI.md` files (one per plugin directory). A distribution question arose: should these files be included in the extension that end-users install, or omitted?

---

## What These Files Are

Each `plugins/<name>/GEMINI.md` documents a plugin's agents, commands, and skills in a structured reference format. They are generated by `tools/generate_plugin_gemini_md.py` from the plugin's actual frontmatter.

---

## The Loading Behavior (Verified)

**Primary source:** [Gemini CLI Extension Reference](https://geminicli.com/docs/extensions/writing-extensions/) — "If a `GEMINI.md` file is present in your extension directory, then that file will be loaded."

This is extension root only. Subdirectory scanning is a project-level feature:

**Primary source:** [Gemini CLI Configuration](https://google-gemini.github.io/gemini-cli/docs/get-started/configuration.html) — "The CLI also scans for the configured context file in subdirectories below the current working directory."

"Current working directory" is the user's project, not `~/.gemini/extensions/<name>/`. Extension subdirectory scanning does not apply to installed extensions.

**Conclusion:** For users who do `gemini extensions install <url>`, only the root `GEMINI.md` loads. The 79 per-plugin files are never auto-loaded.

---

## Options Considered

### Option 1: Remove per-plugin files from extension distribution

Ship only the `commands/` TOML files and root `GEMINI.md`. Remove `plugins/*/GEMINI.md` entirely or add them to a `.geminiignore`.

**Rejected because:**
- Gemini CLI has no `.geminiignore` equivalent for extension files — all files in the extension repo are included in the install
- The files serve developer and power-user workflows (cloning the repo and running `gemini` from within it triggers subdirectory scanning natively)
- The generator script means maintenance cost is near-zero: re-run when plugins change
- "Dead weight" framing overstates the harm: 79 markdown files add negligible overhead

### Option 2: Include all 79 via `@./` in root GEMINI.md

Add `@./plugins/<name>/GEMINI.md` references to the root GEMINI.md to make them accessible to all users.

**Rejected because:**
- Loading all 79 files unconditionally destroys progressive disclosure — the core design principle
- Combined context would exceed practical token limits per session
- Selective loading by plugin would require knowing which plugin the user wants before the session starts, which breaks the auto-activation model

**Primary source on progressive disclosure:** [Gemini CLI Skills](https://geminicli.com/docs/cli/skills/) — skills "activate only when you identify a matching task, saving context tokens."

### Option 3 (Chosen): Keep files in the repo, no changes

Files ship with the extension but serve passive roles:
1. **Developer workflow**: Users who clone and `cd plugins/security-scanning/` get the GEMINI.md scoped context automatically via subdirectory scanning
2. **Human reference**: Readable documentation of each plugin's capabilities directly in the extension directory
3. **Future option**: If Gemini CLI verifies extension-relative `@{path}` resolution, slash commands could reference these files to provide richer context without embedding everything inline

---

## Security and Size Claims (Audited)

A concern was raised that 79 GEMINI.md files create "security and maintenance liability."

- **Security**: Markdown documentation files contain no executable code, secrets, or user data. There is no security surface.
- **Size**: 2,632 lines across 79 files ≈ 158 KB uncompressed. The "~2.6MB" figure circulating in discussion is off by roughly 16×.
- **Maintenance**: Addressed by the generator script. Running `python3 tools/generate_plugin_gemini_md.py` regenerates all 79 files from current plugin frontmatter.

---

## Decision

Keep the per-plugin GEMINI.md files in the extension as-is. Do not add them to root GEMINI.md. Do not remove them from distribution.

## Consequences

- Normal extension users: files are present but dormant (no automatic loading, no harm)
- Developer/power users: auto-loaded when working inside the repo tree
- Phase A slash commands: use self-contained prompts (per separate ADR); these files are not referenced by TOML commands due to the `@{path}` extension-relative resolution limitation
- Future: if extension-relative file inclusion is ever supported or verified, these files become directly valuable to TOML command prompts
